DeckCoach Risk Analysis — Technical Answers
============================================
Prepared: April 12, 2026
App: DeckCoach (formerly DeckRipperHS)
Domain: hs.deck.coach
Platform: macOS native (Swift/SwiftUI)


1. DATA SOURCES
===============

HearthstoneJSON (Card Database)
  Domain: api.hearthstonejson.com
  Data: Card names, stats, text, dbfIds, set, rarity, mechanics
  Method: Static JSON download
  URL: GET /v1/latest/enUS/cards.collectible.json
  Notes: Downloaded once, cached locally for 7 days. HearthstoneJSON is a community-run open-source project (GitHub: HearthSim/hearthstonejson). Not an official Blizzard service.

HearthstoneJSON CDN (Card Images)
  Domain: art.hearthstonejson.com
  Data: Card render images (256x and 512x PNG)
  Method: Hotlinking
  URL: GET /v1/render/latest/enUS/256x/{cardId}.png
  Notes: Loaded on-demand via SwiftUI AsyncImage. Not cached to disk by our code (OS URL cache only). Not bundled in the app.

HSReplay.net (Meta Deck Data)
  Domain: hsreplay.net
  Data: Meta deck lists (card dbfIds + counts), win rates, game counts, archetype names
  Method: Scraping via hidden WKWebView with JavaScript injection
  URLs: hsreplay.net/analytics/query/list_decks_by_win_rate_v2/ and /api/v1/archetypes/
  Notes: Uses WKWebView (real browser engine) to bypass Cloudflare protection. Scraping happens client-side in the macOS app, not on our server.

Together.ai (AI Chat)
  Domain: api.together.xyz
  Data: AI chat completions (DeepSeek-V3.1 model)
  Method: API (paid, with API key)
  URL: POST /v1/chat/completions
  Notes: Legitimate API usage with our paid API key. SSE streaming responses. Proxied through our server so the API key never reaches the client.

Hearthstone Process Memory (Collection Data)
  Source: Local machine, running Hearthstone process
  Data: Card IDs, counts, premium status
  Method: Mach VM APIs (task_for_pid + mach_vm_read)
  Notes: Read-only operation. Never writes to Hearthstone memory or modifies game files. Requires admin password from user.

Google Fonts CDN
  Domain: fonts.googleapis.com
  Data: Philosopher font
  Method: Static download
  Notes: Used on the marketing website only, not in the app.


2. CARD IMAGE PIPELINE
======================

Images are HOTLINKED from art.hearthstonejson.com at runtime.

URL pattern: https://art.hearthstonejson.com/v1/render/latest/enUS/256x/{cardId}.png (or 512x)

How it works:
- SwiftUI AsyncImage loads the URL when a card appears on screen
- The OS handles HTTP caching (memory/URL cache)
- We do NOT explicitly save images to disk
- No images are bundled in the app binary
- No images are downloaded at build time
- No images are hosted on our server

HearthstoneJSON renders these card images from Blizzard's card data. The rendered images contain Blizzard's card art, frames, mana crystals, and attack/health icons. These are Blizzard copyrighted assets rendered and hosted by a third-party community service.


3. USE OF BLIZZARD ASSETS
=========================

Assets displayed (all via HearthstoneJSON, not bundled):
- Card art (illustrations) — displayed in card grid, deck builder, hero picker
- Card frames (rarity borders, card shape) — part of HearthstoneJSON renders
- Mana crystal icons — shown on card images (top-left blue crystal)
- Attack/health icons — shown on card images (bottom corners)
- Card names and text — from HearthstoneJSON card database, displayed in tooltips, search, AI prompt context
- Class names — hardcoded strings (MAGE, WARRIOR, etc.), used for filtering
- Hero skin names — from HearthstoneJSON card database, used in hero picker and AI chat persona

Assets NOT used:
- No Blizzard fonts (we use Philosopher, open-source OFL license)
- No Blizzard sounds or audio
- No Blizzard logos or icons (we use Apple SF Symbols for all UI icons)
- No Blizzard UI elements (our UI is 100% custom SwiftUI)
- The app icon is our own custom witch artwork, not Blizzard art


4. STATS AND ANALYTICS ORIGIN
==============================

Deck win rates: From HSReplay.net, scraped via WKWebView
Deck popularity / games played: From HSReplay.net, same scraping method
Archetype names (e.g. "Herald Shaman"): From HSReplay.net, scraped from /api/v1/archetypes/
Collection ownership percentages: Derived internally — calculated by comparing owned cards against HearthstoneJSON's full card list
"Your Top 3" deck matching: Derived internally — cross-references scraped deck lists against user's owned cards
Card count statistics: Derived internally — counted from extracted collection data

No stats are approximated or fabricated. Win rates and meta data come directly from HSReplay.


5. DECK INPUT AND TRANSFORMATION
=================================

Inputs:
1. Memory scan: raw card IDs + counts from Hearthstone's process memory
2. Deck code import: Base64-encoded Hearthstone deck string (standard clipboard format)
3. AI-generated decks: [DECK]...[/DECK] blocks from DeepSeek-V3.1 responses
4. Net deck import: deck lists (dbfId + count arrays) from HSReplay scraping

Transformations:
- Card IDs resolved against HearthstoneJSON database to get names, images, stats
- Deck codes encoded/decoded using Hearthstone's public Base64 varint format (documented, not proprietary)
- AI deck validation: strip unowned cards, cap copy counts, enforce class restrictions
- Collection stats: aggregate counts, class breakdowns, ownership percentages

Outputs:
- Visual card browser with Blizzard card art (hotlinked, not stored)
- Hearthstone-standard deck code strings (copied to clipboard for pasting into Hearthstone)
- AI chat responses (original text generated by DeepSeek-V3.1 with our custom prompts)
- Derived statistics (ownership %, match counts)

What is original vs copied:
- UI design, layout, all code: 100% original
- AI coaching responses: original (generated by AI with our prompts and player context)
- Card images: copied (Blizzard art rendered and hosted by HearthstoneJSON)
- Card data (names, text, stats): copied (Blizzard game data via HearthstoneJSON)
- Meta deck stats (win rates, popularity): copied (HSReplay data)
- Deck code format: public standard (Blizzard published the format for community use)


6. BACKEND ARCHITECTURE
========================

AI Chat flow (the main server function):
1. App launches, sends POST /register to our server, receives UUID device token
2. User types message in MY HERO chat
3. App sends POST /chat with messages array + device token to deckcoach-api.onrender.com
4. Server validates device token, checks subscription status and rate limit (5 free/day)
5. Server forwards the request to POST https://api.together.xyz/v1/chat/completions with our API key
6. Together.ai streams SSE response, server passes it through verbatim to the app
7. App displays tokens live as they arrive
8. Server increments daily usage counter in SQLite

There is NO hidden scraping on the server. The server is a pure SSE pass-through proxy for Together.ai. All HSReplay scraping happens client-side in the macOS app via WKWebView.

Other server endpoints:
- GET /status: returns subscription/usage data from our SQLite DB
- POST /activate-beta: validates a secret phrase, grants unlimited access
- GET /admin/dashboard: admin-only web dashboard (protected by secret key)
- GET /dl: serves download page and DeckCoach.zip
- GET /: marketing home page (static HTML)
- GET /health: health check for Render.com


7. CACHING AND STORAGE
========================

Client-side (macOS app):
- Card database JSON: ~/Library/Caches/DeckCoach/cards.json, 7-day TTL, re-fetched from HearthstoneJSON
- Card images: macOS URL cache (in-memory, OS-managed), not explicitly cached by our code
- Collection data: /tmp/hearthstone_collection.json, persists until OS clears temp or user re-scans
- User data (favorites, decks, synergies): ~/Library/Application Support/DeckCoach/UserData/, permanent, user-created
- Device token: macOS Keychain, permanent
- Appearance settings: JSON file in UserData directory, permanent

Server-side (Render.com):
- Device records: SQLite on persistent disk, permanent
- Chat usage counts: SQLite on persistent disk, permanent (per-device per-day)
- Subscription records: SQLite on persistent disk, permanent
- DeckCoach.zip build artifact: persistent disk /data/releases/, replaced on each upload

No third-party data is stored long-term on the server. The server never sees card data, images, or HSReplay stats. Those are all client-side. The server only stores device tokens, usage counts, and subscription status.

10. TERMS OF SERVICE AWARENESS
================================

Blizzard Fan Content Policy: Not explicitly reviewed. We should review this — it governs fan-made tools and content.

Blizzard EULA: Not reviewed. The task_for_pid memory reading may violate the EULA's anti-reverse-engineering clause, although we only read data (never write/modify) and the operation is limited to extracting the user's own card collection.

HSReplay.net ToS: Not reviewed. We are scraping their site via WKWebView to bypass Cloudflare. This likely violates their Terms of Service.

HearthstoneJSON: Partially aware. It is an open-source community project (HearthSim), MIT-licensed code. However, the card art it renders is Blizzard's copyrighted material.

Together.ai: Compliant. Using their API with a paid account and valid API key under their standard terms.

Philosopher font: Compliant. OFL (Open Font License) — free for commercial use.

Honest assessment: We have not formally reviewed any Blizzard or HSReplay terms of service. We are operating based on precedent from what other community tools (Hearthstone Deck Tracker, HSReplay itself, Firestone, etc.) have done historically without legal action from Blizzard.


11. COULD THE APP FUNCTION WITHOUT THIRD-PARTY IMAGE SOURCES?
===============================================================

Yes, but with degraded visual experience.

If HearthstoneJSON went away or blocked us:
- Card browser would show text-only cards (names, stats, card text are in the cached JSON database)
- Deck builder would work (card names + IDs are sufficient for building and encoding decks)
- AI Coach would work fully (doesn't need images at all, operates on card text and names)
- Net Decks would work (matches by dbfId, not images)
- Collection scanning would work (reads card IDs from memory, independent of images)

If HSReplay blocked us:
- Net Decks feature would stop working
- Everything else (scanning, browsing, deck building, AI coaching) continues normally

If BOTH were removed:
- The app would still function as a collection scanner + deck builder + AI coach with text-only card display
- The core value proposition (scan your collection, get AI deck advice) would be intact

The app's core functionality (scan, build, coach) does not depend on images. Images make it visually appealing but are not structurally required. The AI coaching layer — our primary monetization — has zero dependency on any third-party visual assets.
